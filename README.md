# Skills

这个仓库用于沉淀可复用的 Codex Skills。目标不是把完整文档搬进 skill，而是把优秀文档里的设计思路、判断框架和执行流程蒸馏成低上下文成本、可触发、可复用的 agent 能力。

## 蒸馏目标

一个好的 skill 应该回答三个问题：

- 什么时候触发：用清晰的 `description` 描述适用任务、领域边界和典型关键词。
- 触发后怎么做：用 `SKILL.md` 给出稳定的工作流、决策顺序、输出结构和风险检查项。
- 细节何时加载：把大段资料拆到 `references/`，只在具体场景需要时读取。

## 文档到 Skill 的转换方法

1. 先读文档结构，不急着摘录内容：识别它解决的问题、面向的角色、核心概念和高频决策。
2. 抽象设计原则：把“为什么这么设计”提炼成可迁移的规则，而不是复述原文。
3. 固化执行路径：把文档里的隐性经验改写为 agent 可执行的步骤、检查表和输出模板。
4. 分离核心与参考：`SKILL.md` 只放必须立即知道的内容；长示例、领域表、状态机和清单放进 `references/`。
5. 保留边界条件：明确不适用场景、常见误区、并发/一致性/安全等风险点。
6. 控制上下文成本：少写背景故事，多写触发条件、步骤、约束和验证方式。

## 推荐目录结构

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── concept-map.md
    ├── workflow-checklist.md
    └── examples.md
```

## `SKILL.md` 编写要点

- `name` 使用短横线命名，保持稳定、语义明确。
- `description` 写给触发器看，覆盖使用时机、任务类型和领域关键词。
- 正文写给 agent 看，优先包含工作流、判断标准、输出形状和红旗清单。
- 引用资料时说明“什么时候读哪个文件”，避免让 agent 一次性加载全部参考内容。
- 不复制大段来源文档；只保留经过重组后的方法、约束和可执行知识。

## 好 Skill 的判断标准

- 触发准确：用户不需要点名 skill，也能在相关任务中自动命中。
- 输出稳定：不同任务能产出结构一致、可落地的结果。
- 细节可追溯：复杂知识能在 `references/` 中按需找到。
- 上下文节省：不把原始长文档塞进主说明。
- 可维护：新增案例或规则时，能放到合适的参考文件而不破坏主流程。

## 当前 Skill

- `ecommerce-design/`：面向电商领域需求分析、领域建模、API/DDL、状态机和风险检查的设计 skill。
- `content-to-skill/`：根据技术博客、文章、文档链接或长文本内容，萃取生成完整可用 Codex Skill，并内置溯源、质检、触发验证与维护规则的核心 skill。
- `insurance-system-design/`：根据有赞《保险系统的分析与设计》萃取的保险系统设计 skill，覆盖保险产品、保单、理赔、渠道、计费对账与风控。

## 维护约定

- 新增 skill 前先明确来源文档的核心价值和适用边界。
- 更新 skill 时优先修改稳定流程；仅当细节会被频繁用到时再提升到 `SKILL.md`。
- 大段领域知识、示例和表格默认放入 `references/`。
- 不在 skill 内添加与执行无关的说明文档，避免干扰 agent 使用。
